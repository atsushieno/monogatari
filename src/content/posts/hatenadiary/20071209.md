#### Mono.Addins

再びLluisによる解説。mono-addinsの設計について。

- 伝統的な手法 と その解決方法:
  - app.exe -(reflection)-> addin1.dll -> lib.dll
  - looking for classes is slow and memory-consuming. => アセンブリ属性
  - all add-ins are loaded even when not needed => アドインマニフェストにロード条件を明記 / アドインのアクティベーションを設定可能
  - multi add-in directories may be required => アドインロケーションを設定可能
  - muli types of extensions (IService, ICommand ... messy) => エクステンションを得るのに共通メソッドを用意
  - some add-ins may depend on others → アドインローダを強化
  - other kinds of extensions: menu, template, help
  - →ホストAPIで登録 / マニフェストで提供
  - アドインは、正しく使おうと思ったら、簡単ではない
- Mono.Addins:
  - MD, #D, Eclipseが背景にある
  - Mono/.NETで動作
  - 以下のような問題を解決:
    - 共通の拡張モデル
    - 利用上の便宜を考えたAPI
    - 遅延ロード
    - アドイン依存処理
    - 柔軟なアドイン検索モデル
    - 拡張の条件付き実行
    - 任意のモジュール
    - ローカリゼーション
- design goals:
  - 自己記述的なアドイン
  - strongly-typed extensions
  - どんな規模のアプリケーションでも実用的
  - スケーラブル
- extension model:

```
  asm1: add-in roots  

    /Service (extension point)  

      - extension node1 (addin), extension node2 (addin) ...  

    /MainMenu  

  asm2:  

    /Service  

    /MainMenu
```

- context of an add-in
  - add-in description
    - XML manifest
    - custom attributes
  - assemblies
    - implement add-in extensions
  - data files
    - deployed together with the add-in
- examples (manifest/custom atts)
- querying an extension point (AddinManager methods)
- custom extension nodes: menu item
- add-in discovery (AddinManager.Initialize() / AddinManager.Registry.Update())
- conclusion:
  - advanced and efficient add-in engine
  - common add-in development model
  - add-in management tools (mautil, online repo, management API)
  - add-in development tools (MD extension, documentation tools)

## パブリックコメント提出活動の成果

さて、出張と体調不良のおかげで2週間ちかく遅れて書いているわけだけど、Mono Summitレポートも終わったことだし、先に進みたい。

というわけで、この前のパブリックコメントの結果が出ていた。  
<http://www.itmedia.co.jp/news/articles/0711/28/news132.html>

7500通の意見のうち8割が「違法サイトからのダウンロード」問題に関する否定的な意見で、その7割がネット上のテンプレに基づくものだったという。多くの人が書いていると思うけど、ネットユーザー側の意見が6000通、うち4200通がテンプレ、それ以外が1800通ということになる。これはけっこう大きな成果だと思う。

（逆に、RIAJらが「動員」をかけた成果が最大(?)1500通くらいあったということでもある。今回はRIAJ会員下のミュージシャンも積極的にコメントを送りたがらなかった人が多いようだし、あまり団結していなかった方だろうから、数としては今後も同じくらいの成果を出していく必要がありそうだ。）

「テンプレ」の中には、[gmail用テンプレート](http://pubcomment.googlepages.com/)や、この問題に関する[先行wiki](http://wiki.livedoor.jp/download11/d/FrontPage)のテンプレも含まれるだろうし、今回は事前に自分のパブリックコメントを公開していた人も多かったから、それをコピーした人もあるいはいることだろう。

MIAU開発プロジェクトWiki2007に[寄せられた](http://dev2007.miau.jp/wiki/wiki.cgi?page=%A5%D1%A5%D6%A5%EA%A5%C3%A5%AF%A5%B3%A5%E1%A5%F3%A5%C8%C4%F3%BD%D0%B5%AD%CF%BF1)提出記録を見ると、ジェネレータの仕組みを使って基本的な枠組みを作った後で、その内容を自分の言葉で書き換えたという人が多いことに気付く。1800人もゼロから書き上げた人がいるとは考えられないので、提出書式と対象ページ番号を埋めるという部分だけでも、ジェネレータやパブコメ素材は、まずまずの成功だったのだろうと思う。

何はともあれ、何らかの形で参加された皆さん、お疲れ様でした。
